# 企业信息调研助手 — 系统级功能验收审计报告

> 审计标准：大厂生产级 Multi-Agent System
> 审计范围：完整代码库逐文件审查

---

## A. 系统是否可用于生产（结论）

### ❌ 不可用于生产

**核心原因**：存在 1 个致命 bug（Synthesizer 未将证据注入 Prompt）+ 3 个关键缺陷（节点无进度事件导致 SSE 近乎空流、同步阻塞异步事件循环、无 token 预算控制），叠加后系统"能跑通但产出不可信"——报告内容是 LLM 凭空生成的，不依赖检索到的证据。

---

## B. 系统存在的 5 个最严重问题

### 问题 1（致命）：Synthesizer Prompt 未注入证据数据

- **模块**：`app/agent/nodes/synthesizer.py` + `app/prompt/prompts.py`
- **具体表现**：
  ```python
  # synthesizer.py 传入了 8 个字段
  report = await chain.ainvoke({
      "research_topic": state["research_topic"],
      "verified_facts": state.get("verified_facts", []),   # ← 传了
      "evidence_pool": state.get("evidence_pool", []),       # ← 传了
      ...
  })

  # 但 SYNTHESIZER_PROMPT 模板中只有 {research_topic}
  # 模板里没有 {verified_facts}、{evidence_pool} 等占位符
  # PromptTemplate(input_variables=["research_topic"]) 也只声明了一个变量
  ```
- **后果**：LLM 在生成报告时看不到任何检索到的证据。Planner → Search → Critic 全流程收集的证据被完整丢弃。最终报告是 LLM 基于 `research_topic` 凭空编造的，相当于把整个检索链路架空了。
- **修复**：Prompt 模板中增加 `{verified_facts}` `{evidence_pool}` `{missing_angles}` 等占位符，并把 `input_variables` 补全。

### 问题 2（关键）：节点无进度事件，SSE 近乎空流

- **模块**：全部 4 个 Agent 节点 + `app/api/server.py`
- **具体表现**：4 个节点函数签名中的 `runtime: Runtime` 参数被导入但未被使用。没有节点调用 `runtime.stream_writer()`。`graph.astream(state, stream_mode="custom")` 收不到任何自定义事件——SSE 在 30-60 秒内流式输出为空，直到 `graph.ainvoke` 结束后一次性返回最终报告。
- **后果**：前端看到的 SSE 流前半段是空白，用户体验是"卡住了 30 秒然后突然弹出一整篇报告"。SSE 的价值完全丧失。
- **修复**：每个节点增加 `writer = runtime.stream_writer; writer({"type":"progress","node":"xxx","status":"running"})`。

### 问题 3（关键）：`web_search.invoke()` 在 async 上下文中同步阻塞

- **模块**：`app/agent/nodes/search.py:16`
- **具体表现**：
  ```python
  raw = web_search.invoke({"query": query_item["query"], "max_results": 5})
  ```
  `web_search` 是被 `@tool` 装饰的同步函数。`.invoke()` 同步调用 Tavily HTTP API（阻塞等待网络 I/O）。而这段代码在 `async def _single_search` 中，外部又被 `asyncio.gather` 并发调度。
- **后果**：`asyncio.gather` 创建了多个协程，但 `web_search.invoke()` 是同步阻塞的——它会在执行时阻塞整个事件循环线程，导致其他协程无法被调度。所谓的"并发搜索"实际上是串行执行（一个阻塞 → 等它完成 → 下一个开始），只是因为都在 asyncio.gather 里所以看起来像并行。性能提升为零。
- **修复**：使用 `await web_search.ainvoke(...)` 或将 Tavily 客户端包装为 `asyncio.to_thread()`。

### 问题 4（重要）：无 Token 预算控制，evidence_pool 无限增长

- **模块**：`app/agent/nodes/search.py:47-49` + `app/agent/state.py`
- **具体表现**：
  ```python
  # search_node: 每轮证据直接追加
  existing = state.get("evidence_pool", [])
  return {"evidence_pool": existing + new_evidence}  # ← 只增不减
  ```
  无证据去重、无数量上限、无 token 计数。3 轮迭代最多产生 3×8×5=120 条证据条目，Critic 和 Synthesizer 的 prompt 会线性膨胀。
- **后果**：第 3 轮迭代时 Context Window 可能超限导致 LLM 调用失败；即使成功，token 成本也远超必要。
- **修复**：增加基于 URL + fact 文本 hash 的去重；Critic 之后裁剪到 Top-N（按 confidence 排序）。

### 问题 5（重要）：Planner 非首轮"空查询保护"存在逻辑漏洞

- **模块**：`app/agent/nodes/planner.py:19-20`
- **具体表现**：
  ```python
  if state.get("iteration_count", 0) > 0 and not missing:
      return {"search_queries": [], "iteration_count": state["iteration_count"]}
  ```
  如果 Critic 未填充 `missing_angles`（例如 LLM 返回的 JSON 缺少该字段），Planner 直接返回空查询。Search 节点拿到空查询后又什么都不做。Critic 下一轮看到同样的证据池，又给出同样的判断。结果：后两轮迭代空转，浪费 2 次 LLM 调用。
- **后果**：系统名义上是"3 轮迭代"，实际上后两轮可能什么都没做。但 iteration_count 照常递增，最终在第 3 轮被强制终止。
- **修复**：非首轮且 missing_angles 为空时，直接 `return {"report_ready": True, "iteration_count": state["iteration_count"]}` 跳过空转。

## C. 每个 Agent 是否"真实有效"（逐个评价）

### Planner
**真实有效度：60%**

✅ 做了的事：首轮确实生成了 3-5 个 search query（LLM 驱动）。
❌ 没做的事：(1) 非首轮空转漏洞（见问题 5）；(2) 无 query expansion——搜索词就是 LLM 生成的原始文本，没有做同义词/多语言扩展；(3) 无 query 数量约束——LLM 可能生成 10+ 个 query，无法控制成本。
**结论**：能拆解问题，但有盲区，不鲁棒。

### Search
**真实有效度：35%**

✅ 做了的事：确实调用了 Tavily API，确实返回了搜索结果。
❌ 没做的事：(1) 并发是假的（见问题 3——同步阻塞）；(2) 无去重——证据池纯追加；(3) 每条搜索结果都调一次 LLM 整理——N 条 query = N 次额外的 LLM 调用——成本翻倍；(4) Tavily API Key 为空时静默失败（返回空列表），不会报错。
**结论**：实现了最低限度的"调 API"，但并发、去重、成本控制全都没做到。

### Critic
**真实有效度：55%**

✅ 做了的事：(1) 确实评估了证据质量并输出 JSON；(2) forced_ready 强制终止机制有效；(3) `iteration_count + 1` 正确递增。
❌ 没做的事：(1) 全部依赖 LLM 判断——没有程序化的质量校验（如 URL 域名白名单、来源权威性评分表、交叉验证算法）；(2) LLM 可能输出 `fact_quality_score: 0.85` 但实际证据全是低质量来源——它是"自我感觉良好"；(3) CRITIC_PROMPT 的 `input_variables=[]` 意味着 prompt 是静态的，所有上下文都通过 `chain.ainvoke(kwargs)` 传入——这在 LangChain 中可行但不规范，容易导致部分数据被静默忽略。
**结论**：评分机制存在但没有硬约束，不能防止 LLM 自欺欺人。

### Synthesizer
**真实有效度：10%**

✅ 做了的事：能生成 Markdown 格式的报告。
❌ 没做的事：**完全没有使用检索到的证据**（见问题 1——致命 bug）。Prompt 模板没有证据注入占位符。报告是凭空编造的。
**结论**：当前实现等同于"把 topic 直接丢给 ChatGPT 写报告"。Planner/Search/Critic 三个 Agent 的工作全部白费。

## D. Workflow 是否真正闭环

| 检查项 | 状态 | 详情 |
|--------|------|------|
| Planner → Search → Critic 主链路 | ✅ 存在 | 硬编码边，确实走这 3 步 |
| loop-back 是否可控 | ⚠️ 部分 | 硬上限有效（3 次），但存在空转漏洞（问题 5） |
| 条件边是否合理 | ✅ 合理 | `report_ready OR iteration >= max_iter → synthesizer` |
| 是否存在假 loop-back | ⚠️ 存在 | 非首轮且 missing_angles 为空时，3 个节点依次执行但没有任何实质工作 |
| 是否存在无效 state transition | ⚠️ 存在 | Planner 返回空 search_queries，Search 什么都不做，Critic 再评一遍同样的数据 |
| 是否存在逻辑断层 | ❌ 存在 | **最大的断层在 Synthesizer**——前面收集的证据不传入 prompt |

## E. 如果这是大厂面试项目，请给评分

### 综合评分：4 / 10

**为什么不是满分：**

| 扣分项 | 严重度 | 扣分 |
|--------|--------|------|
| Synthesizer 证据未注入（致命 bug） | 🔴 Critical | -3 |
| 节点无进度事件，SSE 空流 | 🔴 Critical | -1.5 |
| 同步阻塞伪并发 | 🟡 High | -0.5 |
| 无 token 预算/去重 | 🟡 High | -0.5 |
| Planner 非首轮空转漏洞 | 🟡 Medium | -0.5 |

**扣分理由详细说明：**

1. **-3 分（致命bug）**：这不是"待优化"级别的缺陷。Synthesizer prompt 缺少证据注入意味着整个系统的核心价值主张（"基于检索到的证据生成可靠报告"）是假的。这是面试中会被直接判定"系统不可用"的级别。

2. **-1.5 分（SSE 空流）**：流式输出是面试中重要的工程加分项。4 个节点都不发进度事件 = SSE 退化为一锅端的 HTTP 响应。这段代码"看起来有 SSE，实际没有"。面试官追问"前端怎么展示进度"时无法回答。

3. **-0.5 分（伪并发）**：async/await + asyncio.gather 的代码看起来正确，但底层调用是同步阻塞的。面试官如果问"你怎么验证并发真的有效"，无法给出延迟数据支撑。

4. **-0.5 分（无去重/token预算）**：面试中关于"你怎么控制成本"的问题无法用代码证明。

5. **-0.5 分（空转漏洞）**：展示了对边界情况的思考不够深入。

**缺什么才能到 Production Grade（8+ 分）：**

1. 修复 Synthesizer 证据注入（1 行改 prompt）
2. 4 个节点全部补上 `runtime.stream_writer` 进度事件
3. `web_search.ainvoke()` + 真正的 asyncio 并发
4. evidence_pool 基于 URL hash 的去重 + Top-N 裁剪
5. Tavily API Key 缺失时在 search_tool 层抛出明确异常
6. 增加 `langfuse` 或 `langsmith` 全链路 tracing
7. 增加至少 5 个 pytest（test_planner/mock_tavily/test_critic_forced_ready/test_synthesizer_injection/test_max_iteration）
